# Relatório de Análise e Propostas de Otimização do Banco de Dados (Supabase)

## 1. Visão Geral do Cenário Atual
Atualmente, o banco de dados está consumindo **60% da cota gratuita** do Supabase. O objetivo é reduzir drasticamente esse tamanho sem perder a capacidade de armazenar **4 anos de histórico de vendas** e sem degradar a performance do dashboard.

O sistema atual já passou por uma rodada inicial de otimização ("V2 - Storage Optimized"), removendo descrições longas (como nomes de clientes, produtos e bairros) das tabelas massivas (`data_detailed` e `data_history`), baseando-se em chaves estrangeiras (códigos) para tabelas de dimensão. Isso foi um excelente começo, mas tabelas com milhões de linhas precisam de ajustes ainda mais agressivos a nível de "bytes por linha".

Abaixo detalhamos propostas arquiteturais estruturais avançadas, seus impactos e os riscos envolvidos em implementá-las.

---

## 2. Proposta 1: Fusão das Tabelas de Resumo (`data_summary` e `data_summary_frequency`)
**Contexto Atual:**
Atualmente, o processo de "Refresh Summary" (sincronização de dados de vendas para os dashboards) popula *duas* tabelas agregadas que são altamente redundantes:
- `data_summary`: Agrupa dados financeiros por mês, cliente, vendedor, produto, etc.
- `data_summary_frequency`: Agrupa dados por pedido para contagem distinta e mix, usando estruturas JSONB para produtos e categorias.

**A Solução Proposta:**
Criar uma única tabela (ex: `data_summary_master`) unificando as métricas financeiras (valor, peso, caixas, positivação) e as listas do que o cliente comprou naquele mês.

**Como economizaríamos espaço nessa nova tabela?**
1. **Remoção de Colunas Duplicadas:** Não armazenaríamos ano, mês, filial, códigos de cliente/vendedor duas vezes.
2. **Substituir JSONB por Arrays Nativos (`text[]`):** A estrutura `jsonb_agg(DISTINCT produto)` salva colchetes, aspas e metadados JSON desnecessários. Um array nativo do Postgres (ex: `{707,708}`) gasta muito menos bytes de armazenamento em disco e em memória RAM durante as consultas.
3. **Array de Pedidos (`pedidos text[]`):** Em vez de registrar uma linha para cada pedido de um mesmo cliente no mesmo mês na tabela de frequência, salvaríamos apenas `pedidos: {"123", "456"}` na linha única de resumo daquele cliente/mês. Isso reduz o número de linhas da `data_summary_frequency` de N pedidos para apenas 1 linha por cliente atendido.

### ⚠️ Riscos e Pontos Críticos dessa Junção (O que pode dar errado?)
- **Estouro de Memória (OOM) na Sincronização:** Fazer a junção financeira e o `array_agg(DISTINCT)` em uma única query `INSERT INTO ... SELECT ... GROUP BY` sobre milhões de linhas de vendas é extremamente pesado (requer muita RAM temporária). Isso pode causar um *Statement Timeout (500 Internal Server Error)* ou derrubar o banco durante a recarga de meses passados se os "chunks" não forem processados cliente por cliente ou dia por dia.
- **Diferenças de Granularidade:** Se a interface atualmente consulta o *nível de pedido* para alguma análise de ticket médio (ex: "Qual o valor médio por *pedido*?"), ter a lista de pedidos em array resolve a contagem (`array_length()`), mas perdemos o valor exato isolado de *cada pedido*, tendo apenas o valor total do cliente no mês.
- **Reescrever todas as RPCs:** As funções cruciais como `get_dashboard_filters`, `get_main_dashboard_data`, e `get_frequency_table_data` teriam que ser completamente reescritas para usar operadores de array (`&&`, `@>`) em vez de filtros nativos ou `jsonb ?|`. Qualquer erro lógico aqui pode distorcer métricas como "clientes positivados" ou "mix médio".

## 3. Proposta 2: Substituição ou Remoção de IDs UUID nas Tabelas Massivas
**Contexto Atual:**
Tabelas com milhões de linhas como `data_detailed` e `data_history` (e as de summary) possuem uma coluna `id uuid default uuid_generate_v4() primary key`.

**A Solução Proposta:**
Um UUID ocupa **16 bytes**. Um número inteiro grande (`BIGINT`) ocupa **8 bytes**. Em uma tabela de 5 milhões de linhas de histórico, apenas a coluna UUID ocupa ~80MB, mais ~150MB no índice (B-Tree).

Como seu sistema nunca faz "Updates" pontuais em uma única linha de venda baseada nesse ID (ele sempre apaga e recarrega o mês inteiro via script de sincronização por chaves como `ano-mes`), essas chaves primárias são peso morto.
- Podemos remover a coluna `id` completamente das tabelas `data_detailed` e `data_history`, pois não há dependências de chaves estrangeiras apontando para elas.
- Para as tabelas menores ou necessárias para edição via painel, podemos mudar para `BIGINT GENERATED BY DEFAULT AS IDENTITY`.

### ⚠️ Riscos e Pontos Críticos
- O Supabase Studio (interface web) pode reclamar ou desabilitar edição em lote via grid se uma tabela não tiver chave primária ou identificador único. No entanto, se o seu fluxo é 100% automatizado por scripts/Planilhas, isso é irrelevante.

## 4. Proposta 3: Remoção de Colunas Operacionais Inúteis
**Contexto Atual:**
A tabela `data_detailed` e `data_history` possuem as colunas:
- `created_at` (timestamp, 8 bytes): Data em que o robô inseriu a linha no Supabase.
- `dtsaida` (timestamp, 8 bytes): Quase sempre identica a `dtped`.
- `posicao` (text, média 10-15 bytes): Ex: "FATURADO", etc.

**A Solução Proposta:**
Excluir (`DROP COLUMN`) essas três colunas das tabelas de histórico e detalhes. Só aqui, economizamos de **26 a 30 bytes por linha** (cerca de 150MB a 200MB salvos no total dependendo do número de linhas).

### ⚠️ Riscos e Pontos Críticos
- É necessário garantir 100% de que o front-end, o dashboard e os relatórios (ex: painéis administrativos do cliente) nunca consultam "Qual foi a data de saída exata" nem filtram por "status do pedido". Pelo script de SQL atual (dashboards de inovações, filial, caixas), essas colunas são ignoradas na pré-agregação, tornando a remoção segura.

## 5. Proposta 4: Otimização de Índices (Substituição de B-Trees Grosseiros)
**Contexto Atual:**
Existem índices criados em chaves de baixa cardinalidade em tabelas enormes. Por exemplo, índices normais sobre o campo `dtped`. Um índice B-Tree padrão para datas ocupa muito espaço.

**A Solução Proposta:**
Usar índices do tipo **BRIN (Block Range Index)** para a coluna `dtped` em `data_detailed` e `data_history`.
- Índices BRIN não armazenam um ponteiro para cada linha individual. Eles armazenam os valores "mínimo" e "máximo" da data por blocos físicos do disco.
- Um índice BRIN pode transformar um índice de 200MB em um índice de **1MB ou 2MB**, sendo incrivelmente eficiente para consultas onde buscamos janelas de tempo completas (ex: "Traga todas as vendas do mês 03").

### ⚠️ Riscos e Pontos Críticos
- O índice BRIN só é super rápido se os dados estiverem fisicamente ordenados no disco pela data de pedido (clusterizados). Como vocês sobem as planilhas mensalmente/anualmente, naturalmente os blocos são inseridos em ordem cronológica, o que é o cenário perfeito para BRIN. Se subirem "dados misturados de 2021 no meio de 2025", o desempenho dele piora um pouco.

## 6. Proposta 5: Particionamento Físico Nativo (Table Partitioning) do PostgreSQL
**Contexto Atual:**
Existe uma separação manual (`data_detailed` para meses atuais e `data_history` para arquivos antigos). A query de qualquer ano precisa fazer um `UNION ALL` unindo duas tabelas inteiras. O banco não sabe exatamente onde os dados de 2021 estão antes de rodar a query.

**A Solução Proposta:**
Em vez de duas tabelas "soltas", criar uma tabela `sales_partitioned` e particioná-la nativamente por ano (ex: `sales_2021`, `sales_2022`, etc).
- **Vantagem de Custo:** Com partições, o PostgreSQL lê metadados menores na memória RAM (economiza CPU).
- Quando quisermos deletar o ano de "2020", rodamos um simples comando `DROP TABLE sales_2020`, que recupera o espaço em disco **instantaneamente**, diferente do `DELETE FROM`, que deixa buracos invisíveis (bloat) no banco até um processo pesado de `VACUUM` rodar.

### ⚠️ Riscos e Pontos Críticos
- Adicionar particionamento em um banco de produção requer copiar os milhões de linhas existentes da `data_history` para as novas partições. Esse processo (se não for feito de forma correta) pode gerar um "pico" de consumo no armazenamento do Supabase (para caber os dados originais + a cópia) e estourar o limite provisoriamente.
- O código de `worker.js` (Upload do frontend) terá que ser alterado para não se preocupar mais com `data_history` vs `data_detailed`, apenas enviando para a tabela principal e o Postgres resolve o roteamento.

## Conclusão e Próximos Passos
As maiores vilãs de consumo de banco na cota gratuita são **tabelas auxiliares redundantes**, **colunas UUIDs/Datas não utilizadas** e **índices inchados (B-Trees)**.

Implementar as Propostas 1 a 4 trará uma redução imediata estimada de **30% a 45% do armazenamento atual**, garantindo anos extras de vida útil para o seu banco na cota Free.
