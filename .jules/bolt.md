## Learnings - Bug Fix on RPC Empty Arrays
The frontend sent missing or empty array parameters as empty arrays or omitted them entirely when calling supabase.rpc for PostgREST endpoints. This caused PGRST202 errors because the PostgREST function signature requires exact argument names matching the SQL schema, and empty arrays can sometimes fail type resolution if the RPC parameter strictly expects a text array explicitly formatted or if keys are entirely omitted from the JS object due to undefined values. I fixed this by ensuring every required parameter exists in the payload, using null as fallback for missing arrays/values.
Also, when extracting data from month dropdowns built with 1-based indexing, avoiding an additional parseInt is critical to prevent querying the database with invalid months which results in extreme database timeouts due to impossible WHERE conditions.

## 2024-05-24 - Optimized getVal in processLojaPerfeita
**Learning:** Checking for new keys inside a hot loop (like `getVal` in `processLojaPerfeita`) with a `for...in` over the object's keys on every cell lookup is a significant CPU bottleneck. Iterating all keys of an object multiple times for hundreds of thousands of rows causes overhead, and creating arrays with `Object.keys()` causes memory allocation/GC overhead.
**Action:** To optimize repetitive object key lookups with fuzzy matching, I cached matched keys in a `Map` (`matchedKeysCache`) and tracked known/missing keys using `Set`s. Most importantly, I tracked the `lastProcessedRow` reference to avoid running the `for...in` key discovery loop redundantly on the same row across consecutive column lookups. This reduces complexity from O(Cols * Keys) per row to O(Keys) per row for new key discovery.
